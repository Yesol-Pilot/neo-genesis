## 1. 붙여넣기용 Deep Research 요청문

```text
3. 수집한 자료에서 주요 사실과 데이터 포인트를 요약한다.
   - 조회수, 좋아요, 댓글, 길이, 업로드 시점, 첫 3초 구성, 포맷, 자막 길이, CTA, 정책 제약, 수익화 조건을 표로 정리하라.
   - 데이터가 없거나 공개되지 않은 항목은 추정하지 말고 `비공개/확인 불가`로 표시하라.
4. 대안적 가정과 시나리오를 비교 분석한다.
   - 카드뉴스형 유지, 뉴스 해설형 전환, 풍자 캐릭터형 전환, 실사풍 장면 해설형 전환, 혼합형 운영을 비교하라.
   - 각 시나리오별 기대 효과, 리스크, 제작 비용, 자동화 난이도, 수익화 적합성, 플랫폼 정책 리스크를 비교하라.
5. 결과를 명확한 구조로 정리해 보고서 초안을 작성한다.
   - 결론을 먼저 제시하고, 근거·표·스키마·템플릿·구현 우선순위로 이어지는 구조를 사용하라.
   - 일반론이 아니라 자동화 코드에 들어갈 수 있는 점수표, JSON 스키마, 게이트 조건, 실패 시 재시도 규칙까지 작성하라.

중요한 안전 경계:

- AI 생성 이미지/음성/영상의 표시, 공개 인물 묘사, 저작권, 뉴스 클립 사용, 정치/선거 무결성 정책을 반드시 검토하라.

연구 대상:

   - 가능하면 최근 6개월 이내 사례를 우선하되, 구조적으로 중요한 스테디 포맷은 별도 표시한다.

2. 상위 콘텐츠의 공통 성공 문법

   - 질문형, 분노형, 반전형, 증거 제시형, 풍자형, 뉴스 클립형, 캐릭터 진행형, 댓글 배틀형으로 분류한다.

   - 단순 카드뉴스와 실제 성과형 숏폼의 차이를 설명한다.

   - 하드코딩 없이 매일 자동으로 다른 주제/제목/이미지/레이아웃/CTA가 나오도록 설계하라.

최종 보고서 형식:
1. 5줄 이내 핵심 결론
2. 사용자 원문 요청 해석과 핵심 연구 질문
3. 조사 방법과 출처 목록
4. 주요 사실과 데이터 포인트 요약
5. 상위 레퍼런스 표
6. 성공 패턴과 실패 패턴
7. 대안 시나리오 비교
8. 주제 선정 알고리즘
9. 포맷/각본/이미지/TTS/레이아웃 설계
10. 업로드/편성/성과 측정 전략
11. 자동화 파이프라인 산출물 스키마
12. 바로 적용 가능한 7일치 콘텐츠 기획 예시
13. 위험 요소와 금지 규칙
14. 구현 우선순위 P0/P1/P2

보고서 품질 기준:
- 가능한 한 최신 공개 자료와 실제 플랫폼 사례를 사용한다.
- 모든 주요 주장에는 출처 URL을 붙인다.
- 출처가 약하거나 추정인 내용은 `추정`이라고 표시한다.
- 단순 일반론은 피하고, 실제 자동 생성 시스템에 넣을 수 있는 규칙/점수표/스키마/템플릿으로 작성한다.
- 한국어로 작성하되, 이미지 생성 프롬프트 예시는 한국어와 영어를 모두 제공한다.
```

## 2. 리서치 결과를 받을 때 검수 기준

Deep Research 결과물이 아래 기준을 만족하지 않으면 재요청한다.

| 항목 | 통과 기준 | 실패 신호 |
| --- | --- | --- |
| 레퍼런스 | 실제 URL, 플랫폼, 성과 지표, 날짜 포함 | "요즘 인기" 같은 일반론 |
| 한국 맥락 | 한국 정치/시사/뉴스/풍자 레퍼런스 중심 | 미국/해외 정치 일반론 위주 |
| 안전성 | 선거/정치 무결성, AIGC, 저작권, 명예훼손 리스크 분리 | 자극성만 강조 |
| 자동화 가능성 | JSON 스키마, 점수표, 탈락 기준 포함 | 사람이 감으로 고르는 방식 |
| 장면 설계 | 컷별 이미지 목적과 프롬프트 규칙 포함 | 촛불, 국회, 법원 배경 반복 |
| TTS | 한국어 발음/호흡/속도/감정 곡선 포함 | 모델명 나열만 함 |
| UI/UX | 모바일 글자 수, 폰트 크기, 안전영역 수치 포함 | "가독성 좋게" 같은 추상 표현 |
| 성과 루프 | 하루 1회 측정, 예약 글 분리, 피드백 반영 규칙 포함 | 반복 측정/조회수 단일 기준 |

## 3. 추가로 요구할 후속 질문

첫 리서치 결과가 약하면 아래를 그대로 후속 질문으로 넣는다.

```text
방금 보고서는 아직 실제 자동 생성 파이프라인에 넣기에는 부족하다. 다음을 보강하라.

1. 한국 정치/시사 숏폼 실제 레퍼런스를 최소 30개로 늘리고, 각 URL과 성과 지표를 표로 정리하라.
2. 각 레퍼런스를 `뉴스 클립형`, `풍자형`, `해설형`, `댓글 배틀형`, `캐릭터 진행형`, `팩트체크형`으로 분류하라.
3. 각 분류별로 첫 3초 훅, 자막 길이, 컷 수, 장면 종류, CTA, 댓글 유도 방식을 비교하라.
4. AI 이미지 생성용 `scene_design_plan.json` 예시를 실제 주제 5개에 대해 작성하라.
5. 모바일 9:16 카드/컷 디자인 시스템을 6개 레이아웃으로 구체화하라.
6. 한국어 TTS 대본 작성 규칙과 나쁜 발음 방지 예시를 30개 이상 제시하라.
7. 최종적으로 `올바른 AiNo` 자동 생성 파이프라인에 들어갈 P0 구현 요구사항만 따로 정리하라.
```

## 4. 리서치 결과를 코드 설계로 변환하는 기준

결과 보고서를 받은 뒤에는 바로 콘텐츠를 만들지 말고, 다음 순서로 변환한다.

1. 레퍼런스 표를 `reference_patterns.json`으로 정규화한다.
2. 주제 점수표를 `publish_quality_strategy.json`에 반영한다.
3. `issue_brief.json`과 `topic_score.json` 생성을 먼저 구현한다.
4. `reference_fit.json`으로 포맷 선택 이유를 저장한다.
5. `editorial_brief.json` 없이는 각본 생성을 차단한다.
6. `scene_design_plan.json` 없이는 이미지 생성을 차단한다.
7. `voice_script.json`에 TTS 발음/호흡/강세를 분리 저장한다.
8. `render_plan.json`에 레이아웃 ID, 안전영역, 전환, 컷 길이를 저장한다.
9. `qa_report.json`에서 텍스트 넘침, 중복 레이아웃, 이미지-대사 불일치, AIGC 표시 누락을 차단한다.
10. `publish_manifest.json`에서 즉시 업로드/예약 업로드/성과 측정 대상을 분리한다.

이 기준을 지키지 않으면 다시 양산형 카드뉴스 생성기로 돌아간다.
